# 产品需求文档（PRD）—— 语音智能日历 Hackathon MVP

## 文档信息

| 项目名称 | 语音智能日历 |
|---------|--------------|
| 版本     | Hackathon MVP |
| 作者     | zhaokaig |
| 更新日期 | 2026-05-30 |

## 1. 项目目标

在两天 Hackathon 时间内，完成一个可演示的**语音优先日历软件**。

用户可以通过网页前端输入语音或文字，系统自动理解日程意图，并完成个人日程的创建、查询、修改、删除。最终演示重点不是复杂功能，而是让观众清楚看到：

1. 用户自然说出日程需求；
2. 系统理解意图并执行；
3. 日历界面立即展示结果。

## 2. MVP 范围

### 2.1 必须完成

- 一个可演示的网页前端；
- 个人日程的创建、查询、修改、删除；
- 语音输入能力；
- 文本输入 fallback，避免现场语音识别失败；
- 用户注册与登录；
- Bearer Token 认证；
- 自然语言解析接口；
- SQLite 本地数据库；
- README 中提供启动方式和演示脚本。

### 2.2 语音支持的核心操作

| 操作 | 示例语音 | 预期结果 |
|------|----------|----------|
| 创建日程 | “明天下午三点和 Alex 开会” | 创建明天 15:00 的会议 |
| 查询日程 | “明天有什么安排？” | 返回明天的日程列表 |
| 修改日程 | “把明天下午三点的会改到四点” | 将匹配事件改到 16:00 |
| 删除日程 | “删除明天下午三点的会议” | 删除匹配事件 |

### 2.3 明确不做

- iOS 原生 App；
- 验证码；
- 复杂权限系统；
- 团队、共享日历、邀请成员；
- 多轮对话和长期记忆；
- 一句话多操作；
- 与 iOS/Android 系统日历同步；
- Push 通知、小组件、语音唤醒；
- 复杂冲突检测和重复日程；
- 循环事件例外日期；
- 单次循环实例的独立修改或删除。

## 3. 产品形态

### 3.1 前端

优先实现网页前端，保证两天内可完成并可稳定演示。

页面包含：

- 日历或日程列表展示；
- 语音输入按钮；
- 文本输入框；
- 当前解析结果和执行结果展示；
- 刷新后的日程列表。

前端不需要精美设计，但需要清楚、稳定、适合现场演示。

### 3.2 后端

后端提供 REST API，负责：

- 用户注册与登录；
- 访问令牌校验；
- 日程 CRUD；
- 自然语言解析；
- 将解析出的操作执行到数据库；
- 返回结构化结果给前端。

建议技术方案：

- Python + Flask 或 FastAPI；
- SQLite；
- Werkzeug password hash；
- OpenAI API 或兼容 LLM API；
- 简单 JSON schema 约束模型输出。

## 4. 核心用户流程

### 4.1 创建日程

1. 用户点击语音按钮；
2. 用户说：“明天下午三点和 Alex 开会”；
3. 前端获得语音转文字结果；
4. 前端调用后端自然语言接口；
5. 后端解析为 `create_event`；
6. 后端创建事件；
7. 前端刷新日程列表。

### 4.2 查询日程

1. 用户说：“明天有什么安排？”；
2. 后端解析为 `query_events`；
3. 后端按日期范围查询事件；
4. 前端展示查询结果。

### 4.3 修改日程

1. 用户说：“把明天下午三点的会改到四点”；
2. 后端解析为 `update_event`；
3. 后端根据时间和关键词查找候选事件；
4. 若只匹配到一个事件，直接修改；
5. 若匹配到多个事件，返回需要用户手动选择；
6. 前端展示修改结果并刷新日程列表。

### 4.4 删除日程

1. 用户说：“删除明天下午三点的会议”；
2. 后端解析为 `delete_event`；
3. 后端根据时间和关键词查找候选事件；
4. 若只匹配到一个事件，直接删除；
5. 若匹配到多个事件，返回需要用户手动选择；
6. 前端展示删除结果并刷新日程列表。

## 5. 功能需求

### 5.1 日程数据字段

用户字段：

- `id`：用户 ID；
- `username`：用户名，唯一；
- `password_hash`：密码哈希；
- `created_at`：创建时间；
- `updated_at`：更新时间。

事件字段：

- `id`：事件 ID；
- `user_id`：事件所属用户；
- `title`：标题；
- `start_time`：开始时间；
- `end_time`：结束时间，默认开始后 1 小时；
- `notes`：备注，可选；
- `recurrence_type`：循环类型，支持 `none`、`daily`、`weekly`、`monthly`；
- `recurrence_interval`：循环间隔，默认 `1`；
- `recurrence_until`：循环结束时间，可选；
- `created_at`：创建时间；
- `updated_at`：更新时间。

### 5.2 认证 API

| 方法 | 路径 | 说明 |
|------|------|------|
| `POST` | `/api/auth/register` | 注册用户并返回访问令牌 |
| `POST` | `/api/auth/login` | 登录用户并返回访问令牌 |

### 5.3 日程 API

| 方法 | 路径 | 说明 |
|------|------|------|
| `GET` | `/api/events` | 查询事件，支持 `date`、`start`、`end` 参数 |
| `POST` | `/api/events` | 创建事件 |
| `PATCH` | `/api/events/{id}` | 修改事件 |
| `DELETE` | `/api/events/{id}` | 删除事件 |
| `POST` | `/api/voice-command` | 处理自然语言指令并执行 |

所有日程 API 都需要携带：

```text
Authorization: Bearer <token>
```

### 5.4 循环事件规则

MVP 支持简单循环事件，用于覆盖“每天站会”“每周例会”等常见场景。

支持：

- 不循环：`none`；
- 每 N 天：`daily`；
- 每 N 周：`weekly`；
- 每 N 月：`monthly`。

查询事件时，后端会在指定时间范围内展开循环实例。修改和删除循环事件时，PR2 只作用于整个系列，不支持单独修改或删除某一次实例。

### 5.5 自然语言接口

请求：

```json
{
  "text": "把明天下午三点的会改到四点",
  "timezone": "Asia/Shanghai"
}
```

响应：

```json
{
  "intent": "update_event",
  "status": "success",
  "message": "已将会议改到明天下午四点",
  "events": [
    {
      "id": 1,
      "title": "会议",
      "start_time": "2026-05-31T16:00:00+08:00",
      "end_time": "2026-05-31T17:00:00+08:00"
    }
  ]
}
```

### 5.6 意图类型

后端只需要支持四类意图：

- `create_event`：创建事件；
- `query_events`：查询事件；
- `update_event`：修改事件；
- `delete_event`：删除事件。

暂不支持闲聊。如果模型无法识别为日程操作，返回友好提示：

> 我现在主要能帮你管理日程。你可以试试说：“明天下午三点开会”。

### 5.7 语音修改规则

语音修改是 MVP 核心功能之一，但需要控制复杂度。

支持修改：

- 标题；
- 开始时间；
- 结束时间；
- 备注。

优先支持以下表达：

- “把明天下午三点的会改到四点”；
- “把明天和 Alex 的会议改成产品讨论”；
- “把今天下午的面试备注改成带简历”。

事件匹配策略：

1. 优先按明确时间匹配；
2. 其次按标题关键词匹配；
3. 若多个候选事件分数接近，不自动修改；
4. 返回候选事件，让前端提示用户选择；
5. Hackathon 演示脚本中优先使用能唯一匹配的指令。

## 6. 非功能需求

### 6.1 演示稳定性

- 所有语音操作都必须支持文本输入 fallback；
- LLM 调用失败时，前端需要展示错误信息；
- Demo 数据可以通过脚本初始化；
- 演示路径需要写入 README。

### 6.2 性能

- 日程查询接口响应时间目标 ≤ 0.5 秒；
- 自然语言接口响应时间目标 ≤ 8 秒；
- 若 LLM 较慢，前端展示 loading 状态。

### 6.3 安全

Hackathon MVP 实现最小账号系统：

- 支持用户名和密码注册；
- 密码使用哈希存储；
- 登录后通过 Bearer Token 访问日程 API；
- 每个用户只能访问自己的日程；
- 不实现验证码、找回密码、角色权限等复杂账号能力。

## 7. 验收标准

项目完成时，需要满足以下演示验收：

1. 可以启动后端和前端；
2. 用户可以注册和登录；
3. 网页能展示当前用户的日程；
4. 输入“明天下午三点和 Alex 开会”后，日程被创建；
5. 输入“明天有什么安排”后，能查询到刚创建的日程；
6. 输入“把明天下午三点的会改到四点”后，日程时间被修改；
7. 输入“删除明天下午四点的会议”后，日程被删除；
8. README 中包含完整演示脚本。

## 9. Commit 与 Pull Request 规划

开发过程需要体现清晰、合理的工程拆分。建议控制在 **5 个 Pull Request**，总计 **13–19 个 commits**。

### PR 1：项目脚手架与后端基础

建议 commits：

3. 添加 README 启动说明草稿。

### PR 2：注册登录与日程 CRUD API

建议 commits：

1. 添加认证与事件数据模型；

3. 实现循环事件 CRUD 接口；
4. 添加基础错误处理；
5. 补充 PR2 API 设计说明。

### PR 3：自然语言日程指令

目标：完成语音/文本指令到事件操作的核心链路。

建议 commits：

1. 定义 LLM 输出 JSON schema；

3. 实现修改和删除意图解析；
4. 将解析结果接入事件工具执行。

### PR 4：网页演示前端

目标：完成可演示的用户界面。

建议 commits：

1. 创建前端页面布局；
2. 添加日程列表展示；
3. 添加文本指令输入；

5. 接入后端并刷新结果。

### PR 5：Demo 打磨与文档

建议 commits：
